# 番茄钟功能简要

> 最后更新：2026-03-24
> 适用范围：当前 MVP 的番茄钟后端实现与前后端联调

## 1. 当前目标

当前批次的目标不是一次性补完整个未来架构，而是基于现有仓库，把番茄钟的**真实后端能力**补齐到可联调状态：

- 以 `AppController` 作为番茄钟单一事实源
- 由后端统一计算并广播剩余时间
- 通过状态流转驱动 `resting` / `studying`
- 让 UI 停止依赖正式业务之外的假计时真相

## 2. 本批次必须实现的能力

### 2.1 真实计时与恢复
- 持久化当前阶段开始时间
- 持久化当前阶段类型、阶段总时长、运行状态
- App 切后台、恢复、重启后，能重新计算 `remainingSeconds`
- 超时阶段在恢复时能自动推进，不停留在过期状态

### 2.2 业务状态流转
- 冷启动默认：`resting + inactive`
- 默认显示：下一轮专注时长（25 分钟）
- 首次点击开始后进入 `studying`
- 专注自然结束后自动进入 `resting`
- 休息自然结束后：
  - 若未配置循环，则回到待开始状态
  - 若仍有剩余循环，则进入下一轮专注

### 2.3 可配置项
- 专注时长可修改，默认 `1500` 秒（25 分钟）
- 休息时长可修改，默认 `300` 秒（5 分钟）
- 循环次数支持有限整数轮次
- 默认 `null` / 不循环
- 不支持无限循环

### 2.4 历史统计
当前批次只要求满足统计面板和分享卡片的**最小历史摘要**，例如：
- 今日专注总秒数
- 今日完成专注轮次

不要求当前就实现完整历史明细、复杂图表模型或数据库表设计。

## 3. 实现边界说明

### 3.1 关于“简单数据库”
原始需求里写的是“使用简单的数据库实现番茄钟开始时间的持久化储存”，但对当前 MVP，更重要的是**可靠的本地持久化**，而不是先上完整数据库方案。

当前批次允许采用轻量本地存储来保存标量状态，只要能满足：
- 计时恢复
- 配置恢复
- 历史摘要读取

### 3.2 当前不在本批次内的内容
- 对话系统完整实现
- Live2D 正式接入
- 完整历史明细与图表系统
- 云端同步 / 网络接口
- 大规模 repository / model 拆分重构
- UI 视觉重做

## 4. 验收要点

至少应满足以下结果：
- `toggleTimer()` 能正确开始、暂停、恢复
- `resetTimer()` 能恢复一致的默认状态
- 倒计时文本与顶部进度条来自同一套真实状态
- App 切后台再回来后，剩余时间与阶段正确同步
- App 重启后，能基于持久化开始时间恢复
- `fetchHistoryData()` 只更新历史摘要，不影响当前计时流程
